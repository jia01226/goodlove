# 记忆库优化借鉴（Ombre Brain 优化实战文）

> 来源：圈子里一篇 Ombre Brain（红薯 P0lar1s）优化实战文——作者用出一百多个记忆桶后踩坑+改造的全记录。
> 谁读：柯（记忆核设计合伙人）、小克（施工）。结论先行：**方向跟咱一致,好几个坑咱能提前绕开。**

## ② 直接抄的（并进记忆核设计）
1. **🔴 自动备份（最优先,已动工）**：每天把全部记忆导出 JSON、按日期留档,推到一个**单独的私有仓库**。作者真发生过"重要记忆被整桶覆盖"事故,靠备份救回。咱的 memories.db 目前只有服务器一份——**这是全项目最大的单点风险**。小克已写 `backup_memories.py`,部署时挂 cron。
2. **摘要优先加载（省 80% token）**：启动/浮现时只给"每条记忆一行摘要"（主题+情感+时间）,要细节再展开。咱现在是全文塞或 top-k 全文。**记忆核做 Memory Scene 时照这个设计:场景有"一行版"和"全文版",平时用一行版,聊到了才展开。** 直接影响每次对话成本,跟 prompt 缓存是天作之合。
3. **自动沉底 dormant**：30 天没碰 + 重要度低的记忆自动"休眠"——不进检索、不占上下文,但数据还在,一被提到自动唤醒。咱有时间衰减(权重降低),但没有"彻底让路+可唤醒"。记忆多了以后必须有,不然废记忆越积越糊。
4. **精确匹配优先**：搜人名等短关键词时,精确命中排最前,别被语义模糊结果淹没;**中文两字名不能被拆散**（咱 char-bigram 分词正好要注意这个）。
5. **情绪时间线（valence/arousal）**：每次归档记两个情绪数值,攒成"情绪随时间的曲线"。**正好是咱四层架构"状态层"的现成数据结构**,状态层设计时抄它。
6. **合并去重（merge）**：同一件事记了好几条时能合并（内容追加/标签去重/重要度取高）。记忆核后台维护用。

## ③ 坑清单（花钱买不来的,提前绕）
1. **写操作"覆盖 vs 追加"要焊死**：作者的事故就是"改内容=整条替换"导致重要记忆被覆盖。**咱的规矩:记忆更新一律追加或显式确认,重要记忆(承诺/雷区/排他类)禁止整条替换。**
2. **接口签名不同步 = 功能做了没人知道**：他们改了逻辑忘了更新工具注册,反复修三次。咱的对应:改了后端接口,前端/文档/柯的施工说明必须同步——我每次改完全链路测一遍(已经在这么干)。
3. **备份任务别把配置文件一起提交**：git add 范围限定备份目录,不然权限拒收、备份连带失败。写 cron 时注意。
4. **"AI 自我感受"的权重别给太高**：feel 桶权重 50 压过正常记忆,降到 15 才平衡。对应咱的:柯日记/梦这类"他自己的独白"入记忆库时,权重要压着点,别让"他的感受"淹了"你们的事"。
5. **别名要在所有读写路径上生效**：同一个人多个称呼(佳佳/崽崽/老板)得统一映射,而且每条读写路径都得挂,漏一处就时灵时不灵。

## ④ 明确不抄的
- 记忆桶/MCP 那套架构本身——咱的记忆是 SQLite+md,不引入桶系统,思路吸收、结构不搬。
- GitHub Actions 定时备份——咱直接在 VPS 上 cron,更简单,不依赖 Actions。
