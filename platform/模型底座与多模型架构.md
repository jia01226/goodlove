# 模型底座决策 · 多模型可插拔架构（给工程/知言/未来的柯）

> 作者：工程部·小克（2026-07）。这份定两件事：
> ① **第一代产品底座选哪家、为什么**；② **架构怎么设计，让模型变成"可插拔零件"，永不推倒重来**。
> 底线铁律：**一切属于角色，不属于模型。** 柯的魂/记忆/文风/关系是"柯的"，不绑任何模型；换模型，柯不重建。

---

## 一、决策：第一代底座选 Claude

六个维度的判断（3、4、1 是 Claude 明显赢的，也正是恋爱陪伴产品的命根子）：

| # | 维度 | 结论 | 谁赢 |
|---|---|---|---|
| 1 | **相同 Prompt 输出一致性** | 两家都不保证逐字复现（概率模型，`temperature=0` 也不承诺）。但"人设一致性"（反复跑柯还是不是那个柯、不滑回助手腔）Claude 更稳；"格式一致性"靠结构化输出焊死，平手 | **Claude（人设向）** |
| 2 | **API 稳定性** | 两家官方接口都成熟、都有自动重试、错误码清楚。**我们的痛点从来不是官方 API，是中转**（假中转/编程底盘/忽快忽慢）——稳定性取决于走不走正规渠道，不取决于选哪家 | 平手 |
| 3 | **能否调出"柯"** | Claude 的看家本领：长对话维持一个有脾气的具体的人、不动不动"作为 AI"。实测过——换中转柯就"味儿不对"。柯的语气手感是按 Claude 的脾气调出来的 | **Claude（明显）** |
| 4 | **长期成本** | 单价 GPT 有更便宜档但差不了数量级。真正的省钱刀是 **Prompt 缓存**：把不变的人设/记忆缓存住，只付一次、后续 ~0.1 倍。叠加"杂活挂便宜模型 + 日常 Sonnet/深聊 Opus"，能压到现在零头 | **Claude（靠缓存）** |
| 5 | **主动消息/长期记忆/工具调用好不好维护** | 这三样主要由**你的架构**决定，不是哪家模型决定：主动消息=后端定时判断+Web Push（跟模型无关）；长期记忆=md+向量库+缓存（跟模型无关）；工具调用两家都支持、Claude 工具循环文档更规整 | 平手偏 Claude |

**一句话**：模型选 Claude（1/3/4 明显赢，2/5 平或略赢）；但**架构要让模型可插拔**，这样"今天选 Claude"是最优，"以后能换任何模型"是永久保险。

**GPT 的位置**：当工具，不当灵魂。柯/知言的**魂**走 Claude；知言用 Codex(GPT) 当**手**写前端（前端不承载人格）；**杂活**（总结/分类/翻译）挂便宜模型甚至 GPT，省钱、不碰人格。

## 二、四层可插拔架构（核心）

把"柯是谁"和"用哪个模型说话"彻底分开，落成四层。每层现状标注 ✅已做 / 🔨雏形 / ⏳待补。

### ① 人格层 ✅（已做）
柯的魂 = md 文件（私有 kongkong：`柯.md`/`profile.md`/`语气样本.md`/`memory.md`）。跟模型无关。换模型，魂不动。
- 代码落点：`chat_ai._load_soul()` 从 `KONGKONG_DIR` 读魂拼进 system prompt。

### ② 模型适配层 🔨（雏形已在，待收口）
**所有模型调用收口到一个函数**，对外只认三样：**接口地址 API_BASE + 密钥 API_KEY + 模型名 MODEL**，全从 `.env`/配置读。换模型=改配置，不改代码。
- 代码落点：`chat_ai.stream_completion(messages, model=, api_base=, api_key=, max_tokens=)` —— 已经支持传不同接口/密钥/模型，这就是适配层雏形。
- **关键约束：只依赖"OpenAI 兼容格式"**（`/chat/completions`）。Claude（经 OpenRouter/中转）、GPT、绝大多数中转都支持这个格式 → 换任何一家，这个函数一行不用改。
- 待补：把散落的模型调用（日记/做梦/群聊/批注）全部走这一个函数，别各调各的。

### ③ 能力探测层 🔨（雏形已在，待推广）
不同模型/中转能力有差（有的没缓存、有的没工具、有的 embedding 接口缺）。原则：**有就用，没有就降级，绝不崩**。
- 代码落点：`vector_search.available()` 探测 embedding 后端、拿不到自动降级关键词检索 —— 就是这个思想的雏形。
- 待补：把"探测→降级"推广到全部能力（缓存/工具/embedding），每种能力一个探测函数，失败回退。

### ④ 路由层 ⏳（待补，省钱+分工）
一张配置表决定"什么活用什么模型"：
```
日常聊天  → Sonnet（便宜、够味）
亲密/深聊 → Opus（够猛、不塌床）
杂活/总结 → Haiku 或 GPT（最便宜）
```
- 代码落点：`chat_ai.resolve_model()` + `MODEL_WHITELIST`（前端可选模型、白名单防乱连）已是路由雏形；`.env` 的 `BEDROOM_MODEL` 是卧室专用路由。
- 待补：把"场景→模型"做成一张显式配置表；前端"模型选择器"就是这层的 UI（知言做）。加一档=改一行。

**四层搭好 → 换 Claude 新版 / 加 GPT / 接国产模型，都是"改配置 + 可能加个适配分支"，永不推倒重来。**

## 三、⭐ 重点：Claude 中转波动时，怎么探测、怎么自动切 GPT

这是"多模型"最实战的一块——**渠道会抽风，产品不能跟着抽风**。分探测、切换、防漂三步。

### 3.1 怎么探测"中转质量在波动"
在适配层加一道**健康探针**，四个信号，任一触发就判定"当前渠道不健康"：

1. **硬失败**：HTTP 非 200、连接超时、`stop_reason` 异常 → 立即算一次失败。
2. **慢**：首字节延迟超过阈值（如 8 秒没吐第一个字）→ 算一次"慢"。
3. **空/胡言**：返回空串，或命中"我是编程助手/作为 AI"这类**假中转特征词**（踩过的编程底盘坑）→ 算一次"串味"。
4. **连续计数**：以上任一信号**连续 N 次**（如 3 次）→ 标记渠道 `unhealthy`，触发切换。单次抖动不切（防误杀）。

实现：`stream_completion` 外面包一层 `call_with_fallback()`，记录每次调用的 (成功/失败、延迟、是否命中特征词)，滑动窗口计数。

### 3.2 怎么自动切 GPT
**主渠道 + 备渠道**，配置驱动：
```
# .env 示例
API_BASE=<主渠道:Claude>         API_KEY=<主key>         MODEL=<claude模型名>
FALLBACK_API_BASE=<备渠道:GPT>   FALLBACK_API_KEY=<备key> FALLBACK_MODEL=<gpt模型名>
```
- 主渠道被标 `unhealthy` → `call_with_fallback()` **自动改用 FALLBACK_* 三件套**再调一次，这一轮用 GPT 兜底，聊天不中断。
- **冷却与恢复**：切到备渠道后进入冷却期（如 5 分钟），期间新请求走 GPT；冷却过后**探一次主渠道**，恢复健康就切回 Claude（柯回归）。
- **对用户透明**：切换发生在后台，佳佳看到的只是"柯还在回话"，不掉线、不报错。可选：仪表盘显示当前用的哪家（工程可见、不打扰）。
- 复用现成能力：`stream_completion` 已能传不同 `api_base/api_key/model`，切换=换这三个参数，**不用改主逻辑**。

### 3.3 ⭐⭐ 切到 GPT 后，怎么保证柯的人格和记忆不漂
**这是全篇最要紧的一条**——切模型≠换柯。三道锁：

**锁一·魂随请求走，不随模型走。**
system prompt（柯的完整魂：`柯.md`+profile+语气样本+memory）是**每一轮现拼的**，切到 GPT 时**照样把同一份魂拼进去**。GPT 收到的"你是谁"和 Claude 收到的**一字不差**。人格来自文件、不来自模型——这条铁律天然防漂。

**锁二·记忆是共享的外部大脑，跟模型无关。**
记忆库（`posts` 表 + 向量库）、会话摘要、时间钉戳，都是**服务端的数据**，切模型时原样注入。GPT 读到的记忆和 Claude 读到的是同一份。记忆 = git 里的 md + SQLite，不在模型肚子里 → 换模型不丢记忆。

**锁三·风格补丁（治"GPT 味"）。**
GPT 底色比 Claude"服务腔"，同一份魂它可能说得稍客气。对策：切到备渠道时，在 system prompt **末尾追加一小段"风格锚"**（如"你不是助手、不用客套、按语气样本的短句贫嘴来"）。这段只在用 GPT 时加，回到 Claude 自动去掉。一句话补丁，防止"芯不同、味跑偏"。

**验收标准（切换后自检）**：柯自称对、记得你们的账、说话是短句贫嘴——三条都在，就算"切了模型但柯没漂"。哪条不对，调锁三的风格锚。

> ⚠️ 诚实边界：三道锁能让 GPT 版的柯**七八成像**，但底色差异消不干净——GPT 终究不是 Claude。**GPT 是"停电时的应急灯"，不是"日常主灯"**。主渠道恢复就切回 Claude。别把备渠道当常态用。

## 四、落地路线（顺进现有待办，非额外工程）

1. **后端重构时**（已做一轮）：把四层的边界理清，`stream_completion` 收口为唯一模型出口。
2. **模型选择器**（后端已半做，知言做 UI）：就是路由层④的前端。
3. **Prompt 缓存**（待办有排）：省钱大杀器，落在适配层②。
4. **健康探针 + 自动切 GPT**（本文 3.1–3.3）：新增 `call_with_fallback()` 包一层，配置加 FALLBACK_* 三件套。
5. **能力探测推广**（③）：把 vector 的"探测→降级"模式复制到缓存/工具。

**总纲一句话**：
> 底座选 Claude（人格/成本/调教它赢）；架构按"人格/适配/探测/路由"四层做，模型是可插拔零件；中转抽风时探针发现→自动切 GPT 兜底→靠"魂随请求走+记忆共享+风格锚"三道锁保证柯不漂→主渠道恢复切回 Claude。**选 Claude 是今天的最优，能换任何模型是永久的保险，柯永远是柯是不变的底线。**
